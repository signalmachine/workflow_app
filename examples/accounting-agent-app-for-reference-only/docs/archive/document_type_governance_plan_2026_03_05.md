# Document Type Governance & Manual JE Restriction Plan

**Date:** 2026-03-05  
**Status:** In Progress (implementation started on 2026-03-05)  
**Scope:** Ledger posting policy, document taxonomy, manual JE guardrails, AI routing, migration and rollout strategy.

## 1. Objective

Establish a strict and auditable posting model where:

1. `JE` is reserved for exceptional/manual accounting adjustments.
2. Operational transactions always post with explicit domain document types (not plain `JE`).
3. Every posted ledger entry remains traceable to business intent via a typed document number.
4. Existing workflows migrate safely with low-risk rollout and clear observability.

## 2. Problem Statement (Current State)

The system already supports document-typed postings (`SI`, `PI`, `GR`, `GI`, `SO`, `PO`) and `Ledger.Commit*` always creates a posted `documents` row from `Proposal.DocumentTypeCode`.

Current codebase findings (as of 2026-03-05):

1. Sales customer payment (`RecordPayment`) posts `DocumentTypeCode: "JE"`.
2. Purchase vendor payment (`PayVendor`) posts `DocumentTypeCode: "JE"`.
3. AI default rule still routes many accounting events to `JE` unless clearly sales/purchase invoice.
4. Manual web JE is hardcoded to `JE`, and control-account enforcement is scoped to this path only.
5. `RC` and `PV` are referenced by migration `036_document_numbering_global_uniqueness.sql` and DB health checks, but are not seeded by existing migrations or restore-seed utility.

This causes policy ambiguity: regular business events and true exceptional adjustments can become indistinguishable in reporting/audit.

## 3. Target Policy

### 3.1 Document-Type Rule

All accounting postings must fall into one of two buckets:

1. **Operational / workflow-driven postings**: must use specific document types.
2. **Exceptional/manual adjustments**: may use `JE`, but only under explicit rules.

### 3.2 Baseline Document Taxonomy

Retain existing document types and add payment-specific types:

1. Existing:
- `SO` Sales Order
- `PO` Purchase Order
- `SI` Sales Invoice
- `PI` Purchase Invoice
- `GR` Goods Receipt
- `GI` Goods Issue
- `JE` Journal Entry (restricted)

2. New recommended:
- `RC` Receipt (customer money received)
- `PV` Payment Voucher (vendor payment)

Notes:

1. Naming can be adapted (`CR`, `BP`, etc.), but two dedicated payment types are required.
2. `RC`/`PV` should have `affects_gl = true`, `numbering_strategy = 'global'`, and `resets_every_fy = false` (required to align with global uniqueness policy).

### 3.3 JE Restriction Policy

`JE` is allowed only for controlled scenarios:

1. Opening balances.
2. Accruals and deferrals.
3. Reclassifications/corrections.
4. Period-end adjustments.
5. Reversal/rectification entries not tied to standard operational workflows.

`JE` is not allowed for:

1. Sales invoice creation.
2. Purchase invoice recording.
3. Inventory receipt/issue generated by operations.
4. Customer receipt against invoiced order.
5. Vendor payment against invoiced PO.

## 4. Architecture Changes

## 4.1 Domain Policy Component (New)

Add a centralized policy in `internal/app` (or `internal/core` if preferred) to classify posting intent and validate allowed document types:

1. `PostingIntent` enum (example):
- `manual_adjustment`
- `sales_invoice`
- `purchase_invoice`
- `goods_receipt`
- `goods_issue`
- `customer_receipt`
- `vendor_payment`

2. Policy API (example):
- `ValidateDocumentTypeForIntent(intent, docType, source, metadata) error`

3. Source-aware behavior:
- Manual web (`manual_web`) strict earliest.
- AI/chat initially warn, then enforce after migration period.

This avoids scattered hardcoded checks.

## 4.2 Service Flow Updates

1. `internal/core/order_service.go`
- Change `RecordPayment()` from `JE` to `RC`.

2. `internal/core/purchase_order_service.go`
- Change `PayVendor()` from `JE` to `PV`.

3. Keep existing:
- `InvoiceOrder()` uses `SI`.
- PO receipt inventory/service flows use `GR`.
- Shipment COGS uses `GI`.

4. `internal/adapters/web/accounting.go`
- Manual JE form remains `JE` path, but add policy validation message text clarifying restricted use.

## 4.3 AI Behavior Updates

Update `internal/ai/agent.go` prompt/schema guidance:

1. Replace current “otherwise default to `JE`” logic.
2. Instruct model to choose from full document taxonomy.
3. Require clarification if event type is ambiguous.
4. Add explicit mappings:
- Customer payment/receipt -> `RC`
- Vendor payment -> `PV`
- Adjustment/correction/accrual -> `JE`

Also update schema descriptions (`proposalSchema`) so allowed code examples include `RC`, `PV`, `GR`, `GI`, etc.

## 4.4 Reporting and Audit

1. Add/adjust reporting queries to distinguish:
- manual JE population
- operational postings by document type

2. Extend existing manual JE control-account reports to optionally include:
- “non-JE used for control accounts” sanity checks.

3. Add migration-safe dashboard metric candidates:
- `% postings by doc type`
- `JE share of total postings`
- `operational postings erroneously on JE` count

## 5. Database & Migration Plan

## 5.1 New Migration (next sequence)

Create migration `037_document_type_policy_expansion.sql`:

1. Insert new `document_types` rows (idempotent):
- `RC` Receipt
- `PV` Payment Voucher

2. Set consistent flags (`affects_gl`, `affects_ar`, `affects_ap`, etc.).
3. Set numbering policy for `RC`/`PV` to:
- `numbering_strategy = 'global'`
- `resets_every_fy = false`

4. Add optional guard constraint table or config table (recommended):
- `document_type_policies` with columns like:
  - `intent_code`
  - `allowed_document_type`
  - `is_active`

If deferring table-driven policy, keep logic in code first and add table in a later phase.

## 5.2 Data Backfill (Optional but Recommended)

Create migration `038_backfill_operational_je_classification.sql` (safe/no destructive rewrite):

1. Do **not** rewrite historical posted document numbers.
2. Create reconciliation view/table that tags likely historical operational `JE` entries based on idempotency keys:
- `payment-order-%`
- `pay-vendor-po-%`

This gives audit transparency without mutating immutable history.

Numbering note:

1. Migration `036_document_numbering_global_uniqueness.sql` already exists and must remain unchanged.
2. New plan work starts from migration sequence `037+`.

## 6. Phased Execution Plan

3. Confirm no destructive historical mutation policy.

Deliverables:

1. `document_types` contains `RC` and `PV`.
2. New postings can generate document numbers for both types.

## Phase 2: Core Posting Conversion

Deliverables:

1. Update sales payment posting to `RC`.
2. Update vendor payment posting to `PV`.
3. Keep idempotency key formats stable.

Validation:

1. Integration tests assert `reference_id` belongs to `RC-*` / `PV-*` documents.

## Phase 3: Central Policy Enforcement

Deliverables:

1. Introduce centralized `DocumentTypePolicy` validation.
2. Wire validation before `CommitProposal` and `ValidateProposal`.
3. Restrict `JE` by intent and source.

Rollout mode:

1. Add env var: `DOCUMENT_TYPE_POLICY_MODE=off|warn|enforce`.
2. Start with `warn` in non-prod, then prod.
3. Move to `enforce` after two clean cycles.

Validation:

1. In enforce mode, operational flows using `JE` are blocked.
2. Manual adjustment JE still works with proper context.

## Phase 4: AI & UI Alignment

Deliverables:

1. Update AI prompt and schema descriptions.
2. Update UI/helper text for manual JE restrictions.
3. Add clearer validation errors for wrong doc type usage.

Validation:

1. AI proposes `RC/PV` for payment events in integration checks.
2. Manual JE page clearly indicates approved JE use-cases.

## Phase 5: Observability & Reporting

Deliverables:

1. Add policy violation audit event logging.
2. Add reporting endpoints/queries for JE-vs-operational ratio.
3. Add control report for historical operational-like JE entries.

Validation:

1. Dashboards show near-zero new operational JEs after enforcement.

## 9. Rollout & Safety

## 9.1 Feature Flags

Introduce and document:

1. `DOCUMENT_TYPE_POLICY_MODE` (`off|warn|enforce`)
2. Keep existing `CONTROL_ACCOUNT_ENFORCEMENT_MODE` behavior intact.

## 9.2 Deployment Sequence

1. Deploy migration first.
2. Deploy app with policy mode `warn`.
3. Monitor warning volume and top offenders.
4. Fix any remaining emitters.
5. Switch to `enforce`.

## 9.3 Rollback Strategy

1. If enforcement causes disruption, set `DOCUMENT_TYPE_POLICY_MODE=warn` (no rollback migration needed).
2. Keep new document types in DB; they are additive and harmless.

## 10. Acceptance Criteria

Implementation is complete when all are true:

1. Customer receipt and vendor payment no longer post as `JE`.
2. New operational payment doc types (`RC`, `PV`) are used end-to-end.
3. `JE` usage is restricted to manual/exceptional intents in enforce mode.
4. AI proposal logic no longer defaults unknown accounting events to `JE`.
5. Reporting can distinguish manual JE vs operational postings with measurable trend.
6. Full regression suite and verify tools pass.

## 11. Open Decisions

1. Final code names: `RC`/`PV` vs alternatives.
2. Whether to implement table-driven policy in first iteration or code-only first.
3. Whether AI-originated JE restrictions should enforce at same time as manual-web, or one release later.
4. Whether to add approval workflow requirement for all JE in a future phase (outside this plan).

## 12. Recommended Implementation Order (PR Split)

1. PR-1: migration + seed updates + core payment doc-type switch (`RC`, `PV`) + tests.
2. PR-2: centralized policy layer + mode flag + enforcement tests.
3. PR-3: AI prompt/schema update + adapter messaging + reporting enhancements.

This split keeps risk controlled and allows production rollout with `warn` before full enforcement.
